# 优化审计、设计取舍与后续功能脑暴

## 一、原程序中最值得优先处理的问题

### 1. 固件组合键发送阻塞主循环

原实现按下 Ctrl+Alt+键后执行固定 `delay(15)`，随后释放全部按键。单次影响不大，但长按音量、BLE 状态变化、电量轮询和键盘扫描会共享同一个主循环。连续操作时，阻塞会累积并增加按键边界条件。

**处理方式：** 使用固定队列 + `Idle / ComboReleasePending` 两态状态机。按键按下后立即返回主循环，到达截止时间再释放。

### 2. 设置每次变化立即写 NVS

亮度、熄屏时间和开关每按一次就写入 Preferences。正常使用未必很快写坏，但用户快速循环档位时会产生不必要的写放大。

**处理方式：** 只标记 dirty，最后一次调整 2.5 秒后一次性写入全部设置。

### 3. 长期使用 `String` 保存状态文本

短程序通常没有问题，但嵌入式设备长期运行时，反复拼接和替换动态字符串可能加重堆碎片。

**处理方式：** 改为固定大小字符缓冲区和 `snprintf`。

### 4. BLE 功率注释与枚举不一致

原代码使用 `ESP_PWR_LVL_P3`，注释却写成 -3 dBm。`P3` 是 +3 dBm；真正的 -3 dBm 是 `ESP_PWR_LVL_N3`。

**处理方式：** 省电模式使用 `N3`，高响应模式使用 `P3`，并在诊断页明确显示。

### 5. 页面路径不一致

服务端固定使用 `public` 目录，但项目页面实际在根目录，面板存在 404 风险。

**处理方式：** 明确采用 `public/index.html + app.js + styles.css` 结构。

### 6. PC 状态先修改、后发送

原桥接在 HID 写入成功前就修改 `muted/input/output/volume`。若设备未连接或写入失败，面板会显示一个从未真正执行的状态。

**处理方式：** 命令写入成功后才提交状态，失败时保留原值。

### 7. 缺少 USB 热插拔恢复

设备拔出或短暂异常后，只有下一次操作才会发现，并且需要用户重新连接。

**处理方式：** 低频枚举当前 HID path，自动检测拔出和重新插入；手动断开时停止自动重连。

### 8. 每秒 HTTP 轮询

本地每秒一次并不算重，但页面长期开着时是不必要的固定唤醒。

**处理方式：** SSE 事件推送；只在 SSE 异常时以 3 秒轮询降级。

## 二、已实现的省电策略及其逻辑

### 安全且默认启用

1. 关闭 Wi-Fi、音频、IMU、LED 和外接显示初始化。
2. CPU 默认 80 MHz。
3. BLE 默认 -3 dBm。
4. BLE 广播间隔 100–150 ms。
5. 屏幕默认低亮度并自动熄屏。
6. 熄屏后主循环延时增加。
7. 熄屏后电池轮询由 30 秒降为 120 秒。
8. UI 刷新合并，避免同一瞬间重复全屏绘制。
9. 只有 HID 队列活跃时把循环间隔临时降到 2 ms。
10. 电量低于 10% 时限制背光。

### 有意没有默认启用

#### ESP32 主动 light sleep

理论上可进一步降低 MCU 空闲功耗，但 BLE HID 保持连接、键盘矩阵扫描和不同 Arduino/IDF 配置组合下，主动 `esp_light_sleep_start()` 可能造成连接间隔抖动、首键延迟或重连问题。应在示波器/功耗仪和多台主机上验证后再加入。

#### TFT 控制器 `Display.sleep()`

可比只关背光多省少量功耗，但不同屏幕驱动和批次的唤醒可靠性需要实测，因此保留编译期开关，默认关闭。

#### 40 MHz CPU

可能进一步省电，但对 BLE 回调、SPI 刷屏和键盘扫描余量更小。当前 80 MHz 是更保守的平衡点。

## 三、稳定性策略

### 固件

- 所有时间判断使用无符号差值，能够跨 `millis()` 回绕。
- HID 队列固定 16 项，无法无限增长。
- 队列满时提示用户，不覆盖旧命令。
- 断连时清空队列并释放可能残留的组合键。
- BLE 刚连接后延迟读取 peer address，避免立即访问连接信息。
- 连接状态、电量、设置和 UI 更新互相解耦。
- 不在键盘事件内部执行 Preferences 写入。

### PC 服务

- 所有 HID 命令严格串行。
- 最小命令间隔默认 18 ms，可按实机调节。
- 热键 DOWN 事件做 35 ms 防抖，仍允许固件 90 ms 最快长按重复。
- 设备打开时遍历所有候选 HID 接口，而非只使用 VID/PID 构造器。
- 自动重连失败日志最多每 30 秒重复一次，避免无设备时刷屏。
- HTTP 请求体限制 16 KiB。
- API 只接受已知动作和合法数值。
- 默认只绑定本机回环地址。
- SIGINT/SIGTERM 时关闭键盘监听、SSE 和 HID。

## 四、已加入的实用功能

1. **音量长按加速**：450 ms 后开始重复，持续 2.2 秒后进入快速重复。
2. **音量预设**：适合快速从夜间音量切到正常音量。
3. **安全音量与恢复**：一键降到安全值，再一键恢复。
4. **输入直选**：不需要先知道当前输入。
5. **输出直选**：避免循环时误经过不希望的输出模式。
6. **熄屏首键保护**：可选择首键只亮屏。
7. **省电/响应模式**：近距离长期使用省电，远距离临时提高功率。
8. **诊断页**：显示运行时间、CPU、空闲堆、BLE 功率、HID 队列和 NVS 待写状态。
9. **实时网页面板**：无需每秒轮询。
10. **自动 USB 重连**：设备重新插入后恢复。

## 五、下一阶段可添加的有趣实用功能

以下按价值、风险和前置条件排序。

### A. 高价值、低风险

#### 1. 可编辑快捷键配置

把 `HOTKEY_ACTIONS`、音量步长、预设列表和安全音量放入本地 JSON 配置，并在网页中编辑。需要增加配置校验和原子写文件。

#### 2. 场景宏

例如：

- 夜间：耳放输出 → -50 dB → 取消静音。
- 桌面音箱：LO 输出 → -30 dB。
- 紧急安静：立即静音。

PC 端命令队列已经能自然支持多步骤宏。建议宏之间保留 30–80 ms 间隔并逐步提交状态。

#### 3. 音量渐变

从当前音量平滑升到目标音量，适合早晨或避免突然爆音。要允许用户随时中断，并在新命令到来时取消旧渐变。

#### 4. 系统托盘和开机自启

使用 Electron/Tauri 或原生托盘包装 Node 服务，提供连接状态图标、开机启动和日志入口。核心桥接逻辑无需改变。

#### 5. 键盘锁

例如 `Fn+K` 锁定全部 DX1 控制键，仅保留媒体键；再次长按解除。适合放入口袋或携带时防误触。

### B. 高价值、中等风险

#### 6. 从 DX1 II 读取实际状态

这是解决“推定状态”问题的最佳方案。需要逆向或确认输入报告协议、读取线程、帧校验和设备主动上报机制。完成后可同步旋钮变化、静音、输入和输出。

#### 7. 自动选择正确 HID 接口并记忆 path

首次连接时对候选接口执行无害探测，确认成功后保存 path。风险在于目前协议缺少可靠 ACK/readback，不能仅凭 `write()` 未报错认定接口正确。

#### 8. 多电脑配置

按 Fn 组合键切换不同 BLE 主机或不同控制规则。ESP32 BLE 键盘库当前更偏向单主机 HID，真正多主机切换需要管理 bonds、断开和广播，必须谨慎设计。

#### 9. 低电量策略

低于 5% 时：关闭屏幕、降低广播频率、禁用诊断刷新，并只在按键时短暂唤醒。需要实测 Cardputer 电池百分比的准确性和迟滞。

### C. 实验性、高风险

#### 10. BLE 连接参数自适应

连接稳定后提高 slave latency、放宽 connection interval，可降低无线唤醒频率。但不同 Windows/macOS/Linux 蓝牙栈接受程度不同，设置过激会增加首键延迟或断连。

#### 11. 自动 light sleep

在屏幕关闭、HID 队列为空且没有按键时进入轻睡眠，通过定时器或键盘中断唤醒。需要确认键盘矩阵读取器能否作为可靠唤醒源，以及 BLE 控制器是否在目标 IDF 配置下保持连接。

#### 12. 深度休眠

长时间未连接时进入 deep sleep，按特定键或电源键唤醒。续航提升最大，但会完全断开 BLE，并需要可靠的 GPIO 唤醒设计。

#### 13. 本机声音反馈

可用短促提示音确认静音、输入和输出，但需要重新启用扬声器，会增加功耗并可能影响使用环境。更适合作为可关闭选项。

## 六、建议的实机验证矩阵

### 功耗

分别测量：

1. 高响应模式亮屏。
2. 省电模式亮屏。
3. 省电模式熄屏、BLE 已连接。
4. 省电模式熄屏、BLE 广播等待连接。
5. 低电量自动限亮。

至少记录平均电流、峰值电流和 10 分钟积分电量，而不是只看瞬时值。

### BLE 稳定性

- Windows 11、macOS、Linux 各至少一台。
- 0.5 m、3 m、隔一面墙。
- 连续长按音量 30 秒。
- 熄屏后静置 2 小时再按键。
- 主机睡眠/唤醒。
- 主机蓝牙关闭/重新打开。

### USB HID

- 冷启动时设备已插入。
- 服务运行后插入。
- 命令发送过程中拔出。
- 重新插入后自动恢复。
- 同 VID/PID 多接口设备枚举。
- `PREFIX_REPORT_ID_BYTE=true/false` 两种模式。

### 长时间运行

- Cardputer 连续运行 24–72 小时，观察空闲堆是否持续下降。
- PC 服务连续运行 24–72 小时，观察日志、SSE 客户端和重连定时器。
- 快速切换设置，确认 NVS 写入延迟和重启后恢复。
